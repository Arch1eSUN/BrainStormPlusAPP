# 3.2a Task C1 — Local RLS Verification (Native Postgres)

> Evidence for Task C1 of Sprint 3.2a Chat RLS Tightening. Verifies that
> migration `20260421000000_chat_rls_tightening.sql` produces the intended
> row-level visibility under `auth.uid()`.

## Environment

Task C1 was specified as `supabase start` (Docker-backed full stack). Docker
daemon was unavailable on this host (`docker info` returned "failed to connect
to the docker API"), so verification ran against **native Postgres 15**
(Homebrew service on :5432) with `auth.uid()` stubbed via a session GUC:

```sql
CREATE OR REPLACE FUNCTION auth.uid() RETURNS uuid AS $$
  SELECT NULLIF(current_setting('request.jwt.claim.sub', true), '')::uuid
$$ LANGUAGE sql STABLE;
```

**Coverage**:
- ✅ Policy SQL syntax + planner compatibility
- ✅ RLS evaluation semantics (four-user scenario matrix)
- ✅ `DROP POLICY IF EXISTS` idempotency (re-run safety)
- ⚠️ NOT covered: PostgREST request pipeline, Realtime SELECT replay,
  GoTrue JWT parsing. Full-stack verification (`supabase start`) remains
  open and will run once Docker is available (or folds into C3 staging

## Intermediate bug caught

Initial harness used `SET LOCAL request.jwt.claim.sub = ...` outside a
transaction; Postgres emitted `WARNING: SET LOCAL can only be used in
transaction blocks` and silently ignored the SET, so `auth.uid()`
returned NULL for every role. The resulting behavior — every user saw
only the announcement channel — incidentally **confirmed** that the new
policies correctly deny all non-announcement rows when the session has
no JWT subject. The fix was `SET` (session-scoped) instead of `SET LOCAL`;
the re-run produced the four ✅ assertions above.

## What this proves / doesn't prove

**Proves**:
1. Policy SQL compiles and installs cleanly on a fresh schema.
2. `DROP POLICY IF EXISTS` is idempotent (two successive drops OK).
3. RLS predicate correctly evaluates the three-way union
   `announcement ∪ owned ∪ member-of`.
4. Messages inherit containing-channel visibility via `EXISTS`.
5. Stranger / anonymous access is denied for all non-announcement rows.

**Does not prove**:
1. Realtime subscription replay under RLS (covered by C3 staging shadow).
2. Supabase JWT → `auth.uid()` wiring in production (covered by C2 iOS
   smoke + C3).
3. Performance under real data volume (covered by C3 observation).
4. Interaction with `chat_channel_members` SELECT policy
   (`auth.uid() IS NOT NULL`, a.k.a. debt-10) at scale — stubbed `auth.uid()`
   here behaves identically.
