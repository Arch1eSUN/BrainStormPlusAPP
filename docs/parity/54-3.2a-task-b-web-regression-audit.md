## B2 — Realtime subscription UI dependency check

Full read of `src/hooks/use-realtime.ts` (79 lines) + first 300 lines of
`src/app/dashboard/chat/page.tsx` + grep for all `useRealtimeSubscription`
/ `postgres_changes` consumers.

### Chat page Realtime (`dashboard/chat/page.tsx:200-223`)

Subscribes to `chat_messages` with `channel_id=eq.${selected.id}`;
callback merges INSERT into message list and replaces UPDATE inline. UI
only ever interacts with events for the active channel. Post-RLS, if the
user is viewing a channel they are a member of (or announcement / owned),
they will continue to receive events normally. Access to the channel
itself is already gated server-side by `fetchMessages`'s
`ensureChannelAccess` call, so the narrower RLS only matches that
pre-existing guarantee.

**Verdict**: ✅ safe, no change needed.

### Notification provider global chat listener (`notification-provider.tsx:81-88`)

```typescript
const chatChannel = supabase.channel('global-chat')
  .on('postgres_changes', { event: 'INSERT', schema: 'public', table: 'chat_messages' }, (payload) => {
    const msg = payload.new as any
    if (msg.sender_id === user.id) return
    if (msg.content && msg.content.includes(`@${user.full_name}`) || msg.content.includes(`@${user.display_name}`)) {
      showNotification('新消息提及', `有人提到了你: ${msg.content.substring(0, 30)}...`, '/dashboard/chat')
    }
  }).subscribe()
```

- **Current (`USING (true)`)**: receives INSERT events from every channel
  in the DB. Naive `content.includes('@<name>')` string match triggers
  desktop notification if the message anywhere mentions the user's
  display / full name.
- **Post-RLS**: receives INSERT events only from channels the user is a
  member of (or announcement / owned). Mention detection still runs on
  those events, so user still gets a notification when mentioned in a
  channel they can actually read.
- **Lost behavior**: user no longer gets desktop notifications for
  `@mentions` in channels they are NOT a member of.

**Analysis**: The lost behavior is **undesirable on the current surface**:
clicking the desktop notification links to `/dashboard/chat`, which only
displays channels the user is a member of — so a notification for a
non-member channel is a dead link today. Post-RLS, the notification UX
becomes internally consistent: every desktop mention notification now
corresponds to a message the user can actually open.

**Verdict**: ⚠️ semantic change, but a strict improvement. **No code
change required for 3.2a.** Staging shadow observation (Task C3) should
verify in-member mention notifications still fire.

### Other `postgres_changes` consumers (chat-adjacent, verified unrelated)

### Helper functions

- `normalizeMessage()` (line 206) — pure function, zero DB calls; accepts
  pre-fetched `replyLookup` Map from caller. ✅
- `runMessageSelect()` / `runMessageSingleSelect()` (lines 141, 155) —
  wrapper callbacks that invoke a caller-supplied `buildQuery(db)`;
  retries with LEGACY column selection on error. They don't source their
  own client; every caller passes `adminDb` explicitly. ✅
- Constants `CHAT_MESSAGE_SELECT_FULL` / `CHAT_MESSAGE_SELECT_LEGACY` /
  `LEGACY_CHAT_META_PREFIX` are string literals for column selection; no
  DB interaction. ✅
- Line 3 imports `createClient` (user-role). Used only at line 246 inside
  `requireUser()` for `auth.getUser()` — authentication lookup, not a
  chat-table read. ✅

### Findings

**Zero hidden second-query paths.** Zero legacy fallbacks using a wrong
client. Zero variable shadowing. Reply-to fetch in `fetchMessages`
explicitly re-uses `adminDb`. Type system (`ReturnType<typeof
createAdminClient>` on helper signatures) makes it impossible to pass a
user-role client through `getAccessibleChannelMap` / `ensureChannelAccess`
without a TypeScript error.

## Consolidated verdict

| Read path | Count | Verdict |
| --- | --- | --- |
| Admin-client direct reads | 23 | ✅ Unaffected (service_role bypasses RLS) |
| User-role Realtime subscriptions (filtered) | 1 | ✅ Redundant but compatible |
| User-role Realtime subscriptions (global) | 1 | ⚠️ Semantic narrowing (improvement) |
| User-role direct reads of chat tables | 0 | — |

**Decision**: **Safe to apply migration** `20260421000000_chat_rls_tightening.sql`.

**Required code changes before migration**: **none**.

**Required post-migration observation** (Task C):
1. Local dev DB verification: three users, three scenarios per devprompt
   §3 Task C1.
2. Staging shadow ≥24h: verify notification-provider global chat listener
   continues firing for mentions in in-member channels; verify chat page
   realtime continues delivering active-channel events.
3. iOS end-to-end smoke with unchanged 3.1 client (client-side
   `getAccessibleChannelMap` / `ensureChannelAccess` remains as a defense
   in depth during shadow — removing it is Task D D1 or D2, post-audit).

**Out-of-scope follow-ups surfaced**:
- **debt-10** (membership row leak): `chat_channel_members` SELECT is
  `auth.uid() IS NOT NULL`, so any logged-in user can read all membership
  rows. Currently leveraged by the new `chat_channels`/`chat_messages`
  SELECT policies via their `EXISTS` subqueries; tightening it would
  require either a `SECURITY DEFINER` function or a different design.
  Track separately; not a 3.2a blocker.
- Notification `@mention` detection is naive `string.includes()`;
  probably warrants structured mention handling in a future sprint but
  unrelated to RLS.
