# 1.5 — Projects Server-Side Filter + Membership Scope Foundation

**执行模型：Claude Opus 4.6**

## 0. 轮次定位

`1.4 Projects Filter + Detail Foundation` 已通过 Winston 正式审计：

- `docs/parity/24-winston-audit-1.4.md`

本轮进入 `1.5`，继续收口 **Projects**，但仍然只推进 foundation 级数据正确性与访问边界，不扩散到 full detail parity / CRUD / members UI / AI parity。

`1.4` 已交付：

- iOS Projects list 的 search + status **客户端过滤 foundation**
- iOS Project detail **foundation**
- list → detail 导航

但 `1.4` 后仍存在一个比 detail enrichment 更优先的真实缺口：

1. **iOS Projects list 仍是 client-side filtering，不是 Web 的 server-side filtering semantics**
2. **iOS Projects list 仍未实现 Web 的 membership scoping**
3. **iOS 仍直接 `select()` 全量 `projects`，在数据正确性和权限边界上都落后于 Web source of truth**

所以 `1.5` 的目标不是做更多 UI，而是优先把 **Projects list 的数据获取语义** 向 Web source of truth 收口。

不要扩展到：

- Projects create / update / delete
- member picker / member management UI
- owner profile full rendering UI
- tasks / daily logs / weekly summaries detail enrichment
- AI summary / risk analysis / linked actions / resolution feedback
- OKR / Deliverables / Approvals / Admin 等跨模块工作

## 3. 目标

### 目标 A — Projects list server-side filter foundation

把 `ProjectListViewModel.fetchProjects()` 从 `1.4` 的“全量拉取后本地过滤”推进到更接近 Web source of truth 的 **server-side filter foundation**。

最小要求：

- search text 能参与服务端查询
- status filter 能参与服务端查询
- 不再只依赖当前 `filteredProjects` 做唯一过滤机制
- 查询语义必须对照 Web `fetchProjects()`

实现要求：

1. 必须先核对 Web 端：
   - `if (filters?.status) query = query.eq('status', filters.status)`
   - `if (filters?.search) query = query.ilike('name', `%${filters.search}%`)`
2. iOS 侧可保留轻量 client-side display helper，但**主过滤语义必须下推到 Supabase query**。
3. search 输入变化时，不要做明显粗暴、不可维护的查询风暴。若需要，可做最小 debounce / submit-trigger / 明确 refresh 触发，但必须保持代码清晰可维护。
4. 不要伪造“iOS 已与 Web 完全等价”。如果 `ilike` 行为、大小写、空字符串处理仍有差异，必须记录到 ledger。

### 目标 B — Membership scope foundation

为 iOS Projects list 建立最小可用的 **membership scope foundation**，使非管理员用户不会再无差别读取全部项目。

目标对齐 Web：

- Admin+：可见全部项目
- 非 Admin：只能看到自己在 `project_members` 中关联的项目
- 若用户不是任何项目成员：返回空列表，而不是报错或泄漏全部项目

实现要求：

1. 必须先核对当前 iOS 已有身份来源，例如 `SessionManager.currentProfile` / 当前 auth session。
2. 必须先核对 Web `fetchProjects()` 如何判断 admin 与 membership。
3. iOS 侧可以不完全复刻 Web server action 结构，但必须在真实查询结果上实现同等访问边界。
4. 如果当前 iOS 架构里缺少稳定的 role source / profile source，必须先选择一个最小、真实、可复用的接入点，而不是硬编码 role。
5. 如果 membership scope 需要先查 `project_members` 再查 `projects`，允许两段式查询；但必须保持实现清楚、可审计。
6. 不要为赶进度而引入假的本地 role mock。

### 目标 C — ViewModel / state truth

围绕 1.5 的数据语义调整 `ProjectListViewModel` 与 `ProjectListView`，最小要求：

- `isLoading` / `errorMessage` / empty state 保持可用
- membership 下的 empty state 与普通空列表 / filter no-match 尽量不要语义混乱
- 仍然保留 `1.4` 已有的 search + status 交互体验
- 不要把 detail foundation 搞坏

可以接受：

- 在 list view 中把“本地无匹配”和“服务端返回空”区分得更清楚
- 在必要时对 search 的触发时机做小幅 UI 调整

但不要接受：

- 为了 server-side filtering 把现有 search / filter UI 删掉
- 为了 membership scope 把可工作的 `.projects` 路由打回 backlog

- membership scope 如何落地
- admin / non-admin / no-membership 三种语义如何处理
- 仍缺哪些 Projects parity：detail enrichment、CRUD、members UI、AI、risk 等

默认预期：**`.projects` 仍保持 `.partial`**，除非你能基于真实代码和 Web 对照证明已经完成到可升格的程度。大概率本轮不应升格。
