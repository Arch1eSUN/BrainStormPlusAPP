## 3. 目标

### 目标 A — Nested profile name join foundation on detail sublists

对齐 Web `fetchProjectDetail()` 的 read-only human context：

- tasks: `profiles:assignee_id(full_name)`
- daily logs: `profiles:user_id(full_name)`
- weekly summaries: `profiles:user_id(full_name)`

实现要求：

1. 必须先确认当前 Swift SDK 是否适合稳定 decode nested profile join；如果不适合，不要硬塞进现有 DTO。
2. 可接受两类方案：
   - 方案 1：为 sublists 设计轻量 nested DTO；
   - 方案 2：保持 current summary DTO，不改核心 model，额外对 assignee/user ids 做 batched `profiles.in("id", values: ids)` 查询，再在 view-model 组合 display state。
3. 禁止做 N+1 per-row profile query。
4. 必须是 **batched** 查询，且按 section 或统一 profile cache 聚合。
5. 如果 profile join 失败，对应 row 仍可显示原始内容，不得让整个 section 崩掉。
6. 继续保持 1.6 gate：non-member 不得触发这些 nested profile hydration。

最小完成标准：

- tasks 至少显示 assignee human-readable name（如存在）
- daily logs 至少显示提交人/记录人 human-readable name（如存在）
- weekly summaries 至少显示提交人/记录人 human-readable name（如存在）

### 目标 B — Owner avatar rendering foundation

iOS 已 fetch `avatar_url`，本轮必须把它转化为真实 UI，而不是只保留 name。

实现要求：

1. 至少在以下位置之一完成 owner avatar rendering，优先两处都做：
   - `ProjectCardView`
   - `ProjectDetailView`
2. 可以使用 SwiftUI `AsyncImage` 或项目内已有 image helper；不要发明复杂图片基础设施。
3. 必须有 fallback：
   - 无 `avatar_url`
   - 加载失败
   - URL 非法
   时都不能破 UI。
4. 视觉上保持 foundation 级，不做大改版。
5. 不要引入新的全局依赖。

最小完成标准：

- owner avatar 至少在 list card 或 detail 头部真实渲染一处
- name 与 avatar 同时存在时布局不破

### 目标 C — State truth / failure isolation

最小要求：

- 不回退 1.6 `.denied`
- 不回退 1.7 `tasks` / `dailyLogs` / `weeklySummaries` 已有 section
- profile hydration failure 必须和 base section data failure 分离
- avatar load failure 不得污染主 `errorMessage`
- refresh 仍可用

可以接受：

- 新增 batched profile maps，例如：
  - `taskAssigneesById`
  - `dailyLogUsersById`
  - `weeklySummaryUsersById`
  - 或单个 `profilesById`
- 新增轻量 display helpers / view data

不要接受：

- 因 profile hydration 失败导致 section 全空
- denied path 仍残留 profile maps / avatar state
- 为了 avatar rendering 重写整个 card/detail layout
