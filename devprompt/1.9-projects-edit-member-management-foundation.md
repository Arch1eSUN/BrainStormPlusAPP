## 3. 目标

### 目标 A — Native project edit foundation

对齐 Web 已存在的 `updateProject(...)` 基础能力。

最小要求：

1. iOS 必须至少存在一个真实 edit entry，优先两处都做：
   - `ProjectListView` row actions
   - `ProjectDetailView` header actions
2. 编辑表单至少支持这些字段：
   - `name`
   - `description`
   - `start_date`
   - `end_date`
   - `status`
   - `progress`
3. 保存必须走真实 Supabase update，不允许本地假提交。
4. 编辑成功后：
   - list 刷新
   - 当前 detail 若正在显示同项目，必须同步更新或重载
5. 编辑失败必须有明确错误状态，不得静默失败。

### 目标 B — Member management foundation

对齐 Web `member_ids` 的编辑能力，但先做 foundation，不要求一次性做复杂权限系统。

最小要求：

1. iOS edit flow 中必须能看到当前项目成员状态。
2. 必须能增减成员，沿用 Web 当前 `member_ids` 语义。
3. 必须优先复用已有用户 picker / `profiles` 数据源，不要新造 schema。
4. 必须处理 owner：
   - owner 不应被意外移除
   - 若 Web 语义要求 owner 常驻，iOS 必须与之保持一致
5. member picker 必须是 batched fetch / list，不得按人逐个请求形成 N+1。

### 目标 C — State truth / access / failure isolation

最小要求：

1. 不回退 1.5 list membership scoping。
2. 不回退 1.6 detail membership gate。
3. 不回退 1.7 / 1.8 read-only enrichment。
4. edit / member update 失败不得污染 unrelated read-only sections。
5. 正在保存时必须有明确 loading / disabled state。
6. 非法输入至少要做基础校验：
   - name 不能为空
   - progress 必须被夹在 `0...100`
   - 日期字段允许为空，但不能产生明显错误 payload

### 目标 D — 范围控制

本轮可以做：

- edit entry
- edit sheet / form
- member picker foundation
- update action wiring
- save success refresh

本轮不要做：

- project delete（除非已经极小且顺手完成，但默认不纳入验收主目标）
- create flow 大修
- AI summary
- risk analysis
- linked risk actions
- resolution feedback
- 大型 UI redesign
- schema 变更

### 目标 E — Ledger truth

更新：

- `findings.md`
- `progress.md`
- `task_plan.md`
- 如有必要新增 `docs/parity/33-winston-ready-1.9-notes.md`

必须明确写：

- iOS 现在哪些 edit fields delivered
- member picker / member update 到什么程度 delivered
- 哪些 gap 仍未完成
- `.projects` 是否仍为 `.partial`

默认预期：**`.projects` 仍保持 `.partial`**，除非你能证明 CRUD、members、AI、risk、linked actions、resolution feedback、task_count 等已全部达到 Web parity。正常情况下做不到，所以不要误报。
