## 3. 本轮任务 A — Typed Dashboard Routing

### 3.1 目标

把 Dashboard quick actions / action cards 的核心路由输入从裸字符串迁到 `AppModule`。

当前已知状态：

- `ActionItemHelper.destination(for module: AppModule)` 已存在
- `ActionItemHelper.destination(for title: String)` 仍作为兼容层存在
- `DashboardView` 仍主要通过 title string 调用

本轮目标：

- Dashboard 内部新代码必须以 `AppModule` 为一等输入
- legacy `destination(for title: String)` 可以保留，但不得再是 Dashboard 主路径
- 未迁移模块必须显式进入 `ParityBacklogDestination`

### 3.2 实现要求

1. 检查 `DashboardView` 的 quick action / action item model。
2. 如果当前 action item 只有 `title`，扩展为包含：
   - `module: AppModule`
   - `title`
   - `subtitle` 或 description（如现有需要）
   - icon
   - optional webRoute / status 可从 `AppModule` 派生，不要重复硬编码
3. NavigationLink / destination 必须优先调用：

```swift
ActionItemHelper.destination(for: module)
```

4. 禁止新增新的 fuzzy `contains(...)` routing。
5. 保留 legacy title API 时，必须注释说明：
   - only for backward compatibility
   - new callers must use `AppModule`
6. 若 `AppModule` 缺少 Dashboard 所需 module，必须补齐；但不要引入未验证的 web route。

### 3.3 验收标准

- `DashboardView` 不再以 title string 作为主路由 source。
- `ActionItemHelper.destination(for module:)` 是 Dashboard 主路径。
- `destination(for title:)` 不被 Dashboard 新路径调用。
- 不新增 `PlaceholderDestination`。
- 不新增 `if true`。

## 4. 本轮任务 B — Tasks New Task Create Flow

### 4.1 目标

补齐：

```swift
// TODO: Open New Task Sheet
```

让用户可以在 iOS Tasks 列表里创建 task，并写入与 Web 共享的 Supabase 数据。

### 4.2 Source of Truth 要求

实现前必须确认 Web tasks schema：

- tasks 表真实字段
- required columns
- status enum / allowed values
- priority enum / allowed values
- assignee / creator / project 关系
- org / company / workspace scope 字段
- RLS 依赖的 user/profile 字段

不允许假设字段名。

如果 iOS 当前 `TaskModel` 与 Web schema 不完全一致：

- 优先做最小适配
- 不要大规模 schema 重构
- 在 `findings.md` 记录差异和后续 debt

### 4.3 UI / UX 要求

实现一个原生 SwiftUI New Task sheet，至少包含：

- title（必填）
- description / notes（可选，如 schema 支持）
- due date（可选，如 schema 支持）
- priority（若 schema 支持）
- project picker / assignee picker：本轮可选；如果无法可靠实现，必须明确用当前用户 / 默认空值，并记录 debt
- Cancel
- Create
- loading state
- validation error
- Supabase error display

体验要求：

- title 为空时不能提交
- submit 时禁用重复提交
- 成功后 dismiss sheet
- 成功后刷新列表或通过现有 observable state 插入新 task
- 错误不吞掉，必须显示给用户
- 支持 Dynamic Type 基础可访问性

### 4.4 数据写入要求

优先使用现有 ViewModel / repository 模式。若没有清晰 repository，不要临时把大量 Supabase 逻辑塞进 View；应至少保持：

- View 管 UI state
- ViewModel 管 create action / loading / error
- Supabase client 调用集中在 ViewModel 或现有 service/repository

必须处理：

- auth/session 缺失
- profile/user id 缺失
- Supabase insert error
- reload failure

### 4.5 数据库安全要求

不允许绕过 RLS。

不允许写 service-role key。

不允许硬编码用户 id / org id。

如果当前 session/profile 无法提供必需 scope 字段，则不要伪造；必须：

1. 让 UI 显示无法创建的明确错误，或
2. 使用现有 profile/session 的真实字段，或
3. 记录为 blocker 并停止该子任务
