## 3. 目标

### 目标 A — Native project delete foundation

对齐 Web 已存在的 `deleteProject(id)` 基础能力。

最小要求：

1. iOS 必须至少存在一个真实 delete entry，优先两处都做：
   - `ProjectListView` row actions
   - `ProjectDetailView` header / toolbar actions
2. 删除前必须有明确确认，不允许单击即删。
3. 删除必须走真实 Supabase delete，不允许本地假删除。
4. 删除成功后：
   - list 必须移除对应项目
   - 如果当前 detail 正在显示该项目，detail 必须关闭 / 清空 / 回退到安全状态，不能悬挂已删除项目
5. 删除失败必须有明确错误状态，不得静默失败。

### 目标 B — State truth / destructive-action safety

最小要求：

1. 不回退 1.5 list membership scoping。
2. 不回退 1.6 detail membership gate。
3. 不回退 1.7 / 1.8 / 1.9 的 read/edit/member foundation。
4. delete failure 不得污染 unrelated read-only sections。
5. 正在删除时必须有明确 loading / disabled state，避免重复触发。
6. destructive action 必须：
   - 明确文案（删除项目）
   - 明确不可逆风险提示（若 Web 语义如此则对齐）
   - 至少基础防抖 / disabled state

### 目标 C — Web parity mapping

必须逐行对齐 Web 当前 delete 语义：

`src/lib/actions/projects.ts`

```ts
export async function deleteProject(id: string) {
  await serverGuard()
  const supabase = createAdminClient()
  // project_members cascade-deletes via FK
  const { error } = await supabase.from('projects').delete().eq('id', id)
  if (error) return { error: error.message }
  return { error: null }
}
```

要求：

1. iOS 真实删除目标必须是 `projects` row 本身，不是仅本地移除。
2. 不要额外发明 schema 变更或额外 cascade 逻辑；若数据库 FK 已 cascade，iOS 不要重复造一套。
3. 如果 iOS 因 SDK / UX 需要在本地做状态收口，可以做，但不能改变 server truth。

因为即使本轮补上 delete，AI / risk / linked actions / resolution feedback / `task_count` 等仍未完成。
