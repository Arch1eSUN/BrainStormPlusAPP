## 2. 地面真相：Web RLS vs iOS 直连的致命偏差

Web 所有 chat action 走 `createAdminClient()`（service role），**绕过 RLS**，access control 全在 server action `ensureChannelAccess` / `getAccessibleChannelMap` 里。

iOS 没有 admin client，只能带 user JWT 直连 Supabase。此时实际生效的 RLS：

| 表 | SELECT | INSERT | UPDATE |
| --- | --- | --- | --- |
| `chat_channels` | `USING (true)` —— **任何登录用户可读全部频道** | admin/super_admin/manager role（023） | admin/super_admin/manager 或 created_by（023）/ `auth.uid() IS NOT NULL`（026，仅 last_message） |
| `chat_messages` | `USING (true)` —— **任何登录用户可读全部消息** | `auth.uid() = sender_id` | 未定义（默认 DENY） |
| `chat_channel_members` | `auth.uid() IS NOT NULL` | `auth.uid() = user_id`（只能加自己） | 未定义 |

### 结论（**本轮最核心的架构决策**）

1. **iOS 不能信任 RLS 做频道/消息可见性**。blind `SELECT * FROM chat_channels` 会看到别人私聊的元信息（名字、最后一条消息）；blind `SELECT * FROM chat_messages WHERE channel_id=X` 可以读任意频道的全部消息。
2. **iOS 必须在客户端复刻 `getAccessibleChannelMap` + `ensureChannelAccess`**，否则 UI 会暴露用户本不该看到的数据。
3. send message 的 INSERT 走 RLS `auth.uid() = sender_id`，**安全可以直接调用**。
4. `chat_channels` 的 `last_message` UPDATE 026 策略放宽到 `auth.uid() IS NOT NULL`——**安全可以在 send 后紧跟一次 UPDATE**，不需要 admin client。

### Task B — 重写 `ChatListViewModel`：客户端复刻访问控制

**文件**：`Brainstorm+/Features/Chat/ChatListViewModel.swift`

**动机**：当前 blind `SELECT * FROM chat_channels` 在 RLS `USING (true)` 下 = 读全库频道。必须在客户端执行 Web `getAccessibleChannelMap` 的三段联合：
- `chat_channels.type == 'announcement'`（全员可见）
- `chat_channels.created_by == currentUser.id`（自建）
- `chat_channels` JOIN `chat_channel_members WHERE user_id == currentUser.id`（成员）

**替换整个 ViewModel** 为：

```swift
import Foundation
import Combine
import Supabase

@MainActor
public class ChatListViewModel: ObservableObject {
    @Published public var channels: [ChatChannel] = []
    @Published public var isLoading: Bool = false
    @Published public var errorMessage: String? = nil

    private let client: SupabaseClient

    public init(client: SupabaseClient) {
        self.client = client
    }

    /// Mirrors Web `getAccessibleChannelMap` (src/lib/actions/chat.ts:252-282).
    /// Three parallel queries unioned in-memory — RLS on `chat_channels` is
    /// `USING (true)`, so iOS MUST filter client-side to match Web UX.
    public func fetchChannels() async {
        isLoading = true
        errorMessage = nil
        defer { isLoading = false }

        guard let currentUserId = try? await client.auth.session.user.id else {
            errorMessage = "未登录"
            return
        }

        do {
            async let announcements: [ChatChannel] = client
                .from("chat_channels")
                .select()
                .eq("type", value: "announcement")
                .execute()
                .value

            async let owned: [ChatChannel] = client
                .from("chat_channels")
                .select()
                .eq("created_by", value: currentUserId.uuidString)
                .execute()
                .value

            // Anchor: the membership rows for current user, then resolve channel_ids.
            async let memberships: [ChatChannelMember] = client
                .from("chat_channel_members")
                .select("id,channel_id,user_id,role,joined_at")
                .eq("user_id", value: currentUserId.uuidString)
                .execute()
                .value

            let (anns, ownd, memberRows) = try await (announcements, owned, memberships)
            var memberChannels: [ChatChannel] = []
            let memberChannelIds = memberRows.map { $0.channelId.uuidString }
            if !memberChannelIds.isEmpty {
                memberChannels = try await client
                    .from("chat_channels")
                    .select()
                    .in("id", values: memberChannelIds)
                    .execute()
                    .value
            }

            // Union + dedupe by id
            var seen = Set<UUID>()
            var merged: [ChatChannel] = []
            for ch in anns + ownd + memberChannels {
                if seen.insert(ch.id).inserted { merged.append(ch) }
            }

            // Sort by last_message_at DESC with nulls last (matches Web page.tsx sort)
            merged.sort { (a, b) in
                switch (a.lastMessageAt, b.lastMessageAt) {
                case let (x?, y?): return x > y
                case (_?, nil): return true
                case (nil, _?): return false
                case (nil, nil): return a.createdAt ?? .distantPast > b.createdAt ?? .distantPast
                }
            }

            self.channels = merged
        } catch {
            self.errorMessage = error.localizedDescription
        }
    }
}
```

- 登录为 admin（自己创建过 2 个 group） → 列表显示：自建 2 + 作为成员 N + 全部 announcement，去重无重复。

### Task C — 新建 `ChatRoomViewModel`：`ensureChannelAccess` + fetch + send + realtime

**文件**：`Brainstorm+/Features/Chat/ChatRoomViewModel.swift`（新建）

**职责**：
1. 进入房间前做一次 `ensureChannelAccess` 客户端 gate（防 deep link 直跳到陌生频道）。
2. 初始 fetch：`chat_messages WHERE channel_id = X ORDER BY created_at ASC LIMIT 50`。
3. Realtime 订阅：`postgres_changes` on `chat_messages` filter=`channel_id=eq.{X}`，event=`INSERT`。
4. Send：insert 到 `chat_messages` → insert 返回的 row 直接 append 到本地 array（不等 realtime 回声，避免自己发的消息要延迟 300ms 才显示）→ best-effort UPDATE `chat_channels.last_message` + `last_message_at`（失败不回滚，与 Web `sendMessage` 收尾一致）。
5. cleanup：`onDisappear` / `deinit` 里 `channel.unsubscribe()` + `client.realtimeV2.removeChannel(channel)`。

**实现骨架**（subagent 补完细节）：

```swift
import Foundation
import Combine
import Supabase
import Realtime

@MainActor
public class ChatRoomViewModel: ObservableObject {
    @Published public var messages: [ChatMessage] = []
    @Published public var isLoading: Bool = false
    @Published public var isSending: Bool = false
    @Published public var errorMessage: String? = nil
    @Published public var accessDenied: Bool = false

    private let client: SupabaseClient
    private let channel: ChatChannel
    private var realtimeChannel: RealtimeChannelV2?
    private var realtimeTask: Task<Void, Never>?

    public init(client: SupabaseClient, channel: ChatChannel) {
        self.client = client
        self.channel = channel
    }

    /// Mirrors Web `ensureChannelAccess` (src/lib/actions/chat.ts:284-305):
    /// announcement OR created_by OR membership. Run BEFORE fetch/subscribe
    /// so deep-link into an inaccessible channel reveals nothing.
    public func bootstrap() async {
        isLoading = true
        defer { isLoading = false }

        guard let userId = try? await client.auth.session.user.id else {
            accessDenied = true
            return
        }

        if channel.type == .announcement || channel.createdBy == userId {
            await fetchMessages()
            await subscribeRealtime()
            return
        }

        do {
            let rows: [ChatChannelMember] = try await client
                .from("chat_channel_members")
                .select("id,channel_id,user_id,role,joined_at")
                .eq("channel_id", value: channel.id.uuidString)
                .eq("user_id", value: userId.uuidString)
                .limit(1)
                .execute()
                .value
            if rows.isEmpty {
                accessDenied = true
                return
            }
        } catch {
            errorMessage = error.localizedDescription
            accessDenied = true
            return
        }

        await fetchMessages()
        await subscribeRealtime()
    }

    public func fetchMessages() async {
        do {
            let rows: [ChatMessage] = try await client
                .from("chat_messages")
                .select()
                .eq("channel_id", value: channel.id.uuidString)
                .order("created_at", ascending: true)  // matches Web fetchMessages
                .limit(50)
                .execute()
                .value
            self.messages = rows
        } catch {
            self.errorMessage = error.localizedDescription
        }
    }

    public func sendMessage(_ text: String) async {
        let trimmed = text.trimmingCharacters(in: .whitespacesAndNewlines)
        guard !trimmed.isEmpty else { return }
        guard let userId = try? await client.auth.session.user.id else { return }

        isSending = true
        defer { isSending = false }

        struct InsertPayload: Encodable {
            let channel_id: String
            let sender_id: String
            let content: String
            let type: String
        }

        do {
            let inserted: ChatMessage = try await client
                .from("chat_messages")
                .insert(InsertPayload(
                    channel_id: channel.id.uuidString,
                    sender_id: userId.uuidString,
                    content: trimmed,
                    type: "text"
                ))
                .select()
                .single()
                .execute()
                .value

            // Optimistic append — realtime echo will dedupe in handleRealtimeInsert.
            if !messages.contains(where: { $0.id == inserted.id }) {
                messages.append(inserted)
            }

            // Best-effort last_message / last_message_at update (matches Web sendMessage).
            struct LastMsgPatch: Encodable {
                let last_message: String
                let last_message_at: String
            }
            let iso = ISO8601DateFormatter().string(from: Date())
            _ = try? await client
                .from("chat_channels")
                .update(LastMsgPatch(last_message: trimmed, last_message_at: iso))
                .eq("id", value: channel.id.uuidString)
                .execute()
        } catch {
            errorMessage = error.localizedDescription
        }
    }

    private func subscribeRealtime() async {
        // Supabase Swift SDK v2: channel(name).on(.postgresChanges(...)).subscribe()
        // Authoritative API: https://supabase.com/docs/reference/swift/subscribe
        // Filter syntax mirrors Web's `channel_id=eq.${id}`.
        // ... subagent fills exact v2 API call ...
    }

    private func handleRealtimeInsert(_ row: ChatMessage) {
        // Dedupe: if our optimistic append already landed this id, skip.
        guard !messages.contains(where: { $0.id == row.id }) else { return }
        messages.append(row)
    }

    public func teardown() {
        realtimeTask?.cancel()
        realtimeTask = nil
        if let ch = realtimeChannel {
            Task { await ch.unsubscribe() }
        }
        realtimeChannel = nil
    }

    deinit {
        realtimeTask?.cancel()
    }
}
```

**Realtime 实现指引**（subagent 必须用 Supabase Swift SDK v2 实际 API 替换 placeholder）：

- 用 `client.realtimeV2.channel(name)` 取 `RealtimeChannelV2`。
- 用 `.postgresChange(InsertAction.self, schema: "public", table: "chat_messages", filter: "channel_id=eq.\(channel.id)")` 注册 listener。

```swift
import Foundation

enum ChatDateFormatter {
    /// Mirrors Web page.tsx inline formatter: 今天 HH:MM / 昨天 HH:MM / M/D HH:MM
    static func format(_ date: Date, now: Date = Date()) -> String {
        let cal = Calendar.current
        let hm = DateFormatter()
        hm.dateFormat = "HH:mm"
        if cal.isDate(date, inSameDayAs: now) {
            return "今天 \(hm.string(from: date))"
        }
        if let yesterday = cal.date(byAdding: .day, value: -1, to: now),
           cal.isDate(date, inSameDayAs: yesterday) {
            return "昨天 \(hm.string(from: date))"
        }
        let md = DateFormatter()
        md.dateFormat = "M/d HH:mm"
        return md.string(from: date)
    }
}
```

## 5. 风险 & 边界

### 5.1 Realtime SDK API 是 Swift-SDK-v2 第一次使用

iOS 端没有已 land 的 Realtime 代码。subagent 实现 Task C 的 `subscribeRealtime` 前**必须**：
1. 读 `BrainStorm+-App/Package.resolved` 确认 supabase-swift 版本。
2. 查对应 tag 下的 `Sources/Realtime` 模块 README / public API。
3. **不要** fall back 到 v1 broadcast channel——v2 才有 `postgres_changes` 过滤器。

若 SDK 不支持 filter 语法或支持不完整，**退路是订阅全表**然后客户端过滤 `payload.channelId == channel.id`，但**必须**在 ready-notes 里写明这条降级。

### 5.2 RLS `USING (true)` 是 Web 遗留技术债，不是本轮修复目标

Web 允许所有登录用户 `SELECT` 全部 `chat_channels` / `chat_messages` 行，是因为 Web 通过 `createAdminClient()` 绕过 RLS，server action 层做门禁。从服务端只读的角度这没有**当下的**业务影响，但理论上**任何**能拿到 user JWT 的客户端都能读全量聊天数据。**本轮不修**这条 RLS。修它属于 Web 架构重构（后置到 RBAC Phase 3 或 dedicated security sprint）。iOS 的 defense-in-depth 就是客户端复刻访问控制 + UI 不显示未授权数据。

### 5.3 `chat_channel_members` `USING (auth.uid() IS NOT NULL)` 同样过宽

iOS 查成员表本轮只查 `eq("user_id", currentUserId)`，结果集受 `.eq` filter 约束，RLS 是否 `true` 不影响结果——但如果将来要在 ChatDetailView 展示"本频道成员列表"，就需要在服务端或 RLS 加 `channel_id IN (my accessible channels)`。登记进 debt（3.1-debt-06 —— 3.3+ 补）。

### 5.4 发消息乐观更新 + Realtime 回声的 dedupe

本轮采用**乐观 append + insert 返回真实 row**（用真实 row id 做判重），这样自己的消息立即可见。Realtime `INSERT` 事件到达时 `handleRealtimeInsert` 检查 id 已存在就跳过。理论上如果乐观 append 因为某种原因用了**假 id**（本实现没用假 id，一律等 insert 返回 row），dedupe 就会失败——本实现安全。

### 5.5 本轮不做 announcement "仅 admin 可发"

Web `page.tsx` 在 input 面板上对 announcement 频道加了 admin gate。iOS 本轮让所有人都能发到 announcement（INSERT RLS 允许 `auth.uid() = sender_id`，不会 500）。登记进 debt（3.1-debt-03）。

### 5.6 本轮不做分页

iOS `fetchMessages` limit 50，older messages 不可获取。Web 也是硬 50。Web `page.tsx` 没有 older-load——这条 debt 两端都有。iOS 登记 3.1-debt-04，与 Web 同步时一起解决。
