## 2. 目标

在 iOS `ProjectDetailView` 中提供一个最小写路径 foundation，让用户可以把 Web risk analysis 中的一个可识别 risk item 转为 `risk_actions` 行，或者在 iOS 无法稳定解析 risk item 时提供诚实的“write path blocked” state。

必须先判断真实可行性：

- 如果 Web 的 `syncRiskFromDetection(...)` 依赖 server-only 逻辑、隐藏参数、非公开 schema、或无法从 iOS 安全构造输入，则不要伪实现。
- 如果可由 Supabase direct insert 安全复刻，则实现最小 direct write。
- 如果必须由 Web 暴露 HTTP endpoint 才安全，则本轮改为 endpoint-needed foundation，并明确阻断原因。

---

## 3. 强制设计原则

### 3.A 写入前先 RBAC

必须 mirror Web role gate。

预期 gate（必须重新验证）：

```ts
['super_admin', 'admin', 'hr_admin', 'manager']
```

iOS 必须基于当前 `SessionManager.currentProfile` / `RBACManager.migrateLegacyRole(...)` 判断是否显示/允许写入。

禁止仅依赖服务端 RLS 后置报错作为唯一 UX。

### 3.B 必须有确认 affordance

任何真实写入 `risk_actions` 前，必须有明确确认：

- 要创建什么风险动作
- 来源项目 / risk analysis
- 是否可撤销 / 如何处理
- 写入后会刷新哪些 section

### 3.C 必须隔离写入状态

新增独立状态，不允许污染：

- `errorMessage`
- `summaryErrorMessage`
- `riskAnalysisErrorMessage`
- `linkedRiskActionsErrorMessage`
- `resolutionFeedbackErrorMessage`
- `deleteErrorMessage`
- `enrichmentErrors`

建议新增：

- `riskActionSyncPhase`
- `riskActionSyncErrorMessage`
- `isSyncingRiskAction`

### 3.D 写入后必须刷新相关 read snapshots

成功写入后至少刷新：

- `refreshLinkedRiskActions()`
- `refreshResolutionFeedback()`

不要刷新整个 detail 主态，除非确有必要。

### 3.E 不得泄露或使用 server-only secrets

禁止：

- 读取 `api_keys`
- 调用 `askAI` on-device
- 把 provider key 带到 iOS
- 伪造 Web server action 已被调用

## 4. UI Scope

本轮只允许在 `ProjectDetailView` 的 risk-analysis / linked-actions 附近增加最小 UI。

允许：

- 一个 “Convert to risk action” / “Sync risk action” 按钮
- 一个角色不允许时的 disabled/hidden/hint state
- 一个确认 dialog / confirmation sheet
- 一个写入成功 hint
- 一个 write-path-blocked honest hint（如果 source-of-truth 不足）

禁止：

- 完整 risk action editor
- 完整 action lifecycle manager
- bulk convert
- assignee picker
- due date picker
- risk action detail modal
- governance intervention writer
- resolution close/reopen UI
- i18n 重构
- visual redesign

## 5. 数据可行性审计

实现前必须回答并记录：

1. `syncRiskFromDetection(...)` 需要哪些字段？
2. 这些字段是否存在于 iOS 当前可读数据？

4. 如果从 LLM summary 文本解析，是否可靠？如果不可靠，不允许写。
5. 是否已有 `risk_items` JSONB 或结构化数据可直接用？
6. Direct insert 是否符合 RLS？
7. duplicate prevention 如何做？是否已有唯一约束或 Web-side idempotency？

如果任一关键项不明确，必须停在 blocked foundation，不做伪写入。

## 7. 完成定义

完成必须满足以下二选一：

### Path A — Direct write feasible

- Web source-of-truth 重新核对完成。
- RBAC gate 完成。
- 确认 affordance 完成。
- 写入 `risk_actions` 成功路径完成。
- 写入失败有独立错误面。
- 写入成功后刷新 linked actions + resolution feedback。
- build 通过。
- ledger / ready notes 完成。

### Path B — Direct write not safe/feasible

- 明确证明为什么不能安全 direct write。
- UI 不提供伪写按钮，或只显示 blocked / web-only honest hint。
- ready notes 明确下一步需要 Web endpoint / Edge Function / schema exposure。
- build 通过。
- ledger 完成。
