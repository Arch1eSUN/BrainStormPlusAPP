## 3. 目标

### 目标 A — Projects list filter foundation

把 `ProjectListView` 从“纯列表”推进到最小可用 filter foundation，最小要求：

- 支持 **search text**
- 支持 **status filter**
- 过滤逻辑必须以 Web source of truth 为依据
- UI 采用现有 iOS design tokens / component style
- 不要做过度复杂的 filter 面板

实现要求：

1. 优先复用现有 `Project.ProjectStatus`。
2. filter UI 可以是：
   - search bar / searchable
   - segmented / chip / menu / picker
   但必须是原生 SwiftUI 可维护方案。
3. 如果当前 iOS 侧不能安全复用 Web 的服务端过滤语义，可先做 **客户端过滤 foundation**，但必须在 ledger 中明确记录与 Web 的差距。
4. 不要伪造“已完成 Web 权限/成员范围完全一致”。

### 目标 B — Project detail foundation

建立一个真实可进入的 iOS Project detail foundation，最小要求：

- `ProjectDetailView`
- 如有必要新增 `ProjectDetailViewModel`
- 从已有 list row 点击进入 detail
- 至少展示：
  - project name
  - description
  - status
  - progress
  - start / end date（若有）
  - owner / basic metadata（若现有 source 能安全取得）
- loading / empty-ish fallback / error state

数据策略要求：

1. 必须先核对 Web `fetchProjectDetail()` 的真实 source。
2. 可以只实现 **detail foundation**，不要求把 Web detail 的 tasks / daily logs / weekly summaries / AI sections 全量搬过去。
3. 如果当前 iOS 侧只适合先用现有 `Project` 基础数据做 detail skeleton，也可以，但必须明确写在 ledger：这是 detail foundation，不是 full detail parity。
4. 如果安全且低风险，优先调用真正的 detail fetch，而不是只把 row 数据原样再铺一遍。
