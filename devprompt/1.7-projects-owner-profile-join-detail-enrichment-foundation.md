## 3. 目标

### 目标 A — Owner profile join foundation

让 Projects list / detail 至少能显示 owner 的 human-readable 基础信息，而不是只显示 raw UUID。

Web source of truth：

```ts
select('*, profiles:owner_id(full_name, avatar_url)')
```

实现要求：

1. 必须先核对当前 iOS Supabase Swift 是否能稳定 decode nested `profiles` join。
2. 可以选择以下任一安全方案：
   - 方案 1：新增轻量 DTO / view data model 支持 `profiles:owner_id(full_name, avatar_url)` join；
   - 方案 2：保持 `Project` core model 不变，额外查 `profiles` by `owner_id` 并组合成 detail/list display state；
   - 方案 3：先只在 detail 做 owner profile enrichment，list 保持 raw UUID，但必须说明为什么 list 暂缓。
3. 不要为 owner join 重写核心 `Project` 模型到不可维护状态。
4. 不要让 owner join 破坏 1.5/1.6 的 membership scope 与 detail gate。
5. 如果 owner profile fetch 失败，Project 基础内容仍应尽量可显示，错误要可控。

最小完成标准：

- detail 里不再只显示 raw owner UUID；至少显示 `full_name`，avatar 可选。
- ledger 明确 list owner join 是否完成；如果未完成，记录原因与后续计划。

### 目标 B — Read-only Project detail enrichment foundation

在通过 1.6 detail gate 后，给 `ProjectDetailView` 增加最小 read-only enrichment：

Web `fetchProjectDetail()` 当前包含：

- tasks: up to 50, ordered by `created_at DESC`
- recent_daily_logs: up to 10, ordered by `date DESC`
- weekly_summaries: up to 5, ordered by `week_start DESC`

本轮不要求完整 UI parity，但要求建立真实数据 foundation。

实现要求：

1. 必须在 detail gate 通过后才加载 enrichment。
2. 必须保留 1.6 denied behavior：non-member 不应触发 detail enrichment queries。
3. 可以选择最小 DTO，不必强行复用全 App model；但字段必须来自真实表：
   - `tasks`: `id`, `title`, `status`, `priority`, `assignee_id`
   - `daily_logs`: `id`, `date`, `content`, `progress`, `blockers`
   - `weekly_reports`: `id`, `week_start`, `summary`, `highlights`, `challenges`
4. UI 可以是 compact read-only sections，不要做编辑、跳转、复杂过滤。
5. 如果某个 enrichment query 失败，不要让整个 detail foundation 崩掉；尽量降级为 section error / empty state，并在 ledger 记录。
6. 不要扩展到 AI summary / risk analysis / linked actions / resolution feedback。

最小完成标准：

- detail view 至少展示 tasks / daily logs / weekly summaries 三类中的两类真实数据 foundation；若三类都能低风险完成，优先三类都做。
- 数据必须来自 Supabase 查询，不允许 fake/mock。

### 目标 C — State truth / loading truth

围绕 owner join + enrichment 调整 state，最小要求：

- `isLoading` / `errorMessage` / `.denied` 行为不倒退
- enrichment loading / empty / failure 不与 access denied 混淆
- seeded project detail 仍可快速显示
- refresh 仍可用

可以接受：

- 新增 `ProjectDetailData` / `ProjectOwnerSummary` / `ProjectTaskSummary` 等轻量结构
- 新增 `enrichmentErrorMessage` 或 per-section state

不要接受：

- 一个 enrichment query 失败导致已通过 gate 的 project 基础内容完全消失
- denied 状态仍泄漏 owner / tasks / logs / summaries

### 目标 D — Ledger truth

更新：

- `task_plan.md`
- 如有必要新增 `docs/parity/29-winston-ready-1.7-notes.md`

必须明确写：

- detail enrichment 三类数据各自做到什么程度
- 1.6 access gate 是否保持
- 仍缺哪些 Projects parity：CRUD、members UI、AI、risk、linked actions、resolution feedback 等
- `.projects` 是否仍应保持 `.partial`

默认预期：**`.projects` 仍保持 `.partial`**。
