# 2.4 — Projects Linked Risk Actions Foundation

**执行模型：Claude Opus 4.6**

## 0. 轮次定位

`2.3 Projects Risk Analysis Foundation` 已通过 Winston 独立审计：

- `docs/parity/42-winston-audit-2.3.md`

当前 Projects 模块仍保持 `.partial`。在 2.3 已完成 risk analysis read-only foundation 后，下一项最自然、最窄、最可审计的 detail-surface gap 是：

- **Projects Linked Risk Actions Foundation**

本轮只做：

- 读取并展示与当前项目风险分析相关联的 risk actions 基础列表
- 仅限 detail 页面 risk analysis 区块向下延展的 read-only foundation
- 不做 risk action 创建/编辑/关闭
- 不做 resolution feedback 汇总
- 不做 iOS 侧 generate-risk
- 不做 Web-side `/api/ai/project-risk` HTTP endpoint
- 不做任务、成员、删除、AI summary、task-count redesign

## 2. 目标

在 iOS `ProjectDetailView` 中，为 2.3 的 `riskAnalysisSection` 增加一个紧邻其后的 **Linked Risk Actions** foundation 区块，实现以下能力：

1. **读取与当前项目风险分析关联的 risk actions**。
2. 只做 **read-only 列表展示**。
3. 明确区分：
   - 还没检查 / 没有风险分析 / 有风险分析但无 linked actions / 读取失败 / 读取成功
4. 保持与 2.3 一致的诚实文案：
   - iOS 当前只读，不支持“转为风险动作”写入动作。
5. 严格隔离错误面：
   - linked actions 失败不能污染 detail 主态 / risk analysis / AI summary / enrichment / delete。

---

## 3. 实现约束

### 3.A 数据来源与 source of truth

必须先基于 Web 重新确认真实链路。

预期真实情况（必须重新验证，不可照抄本段当事实）：

- Web 风险分析结果写入 `project_risk_summaries`
- linked risk actions 通过 `ai_source_id = project_risk_summaries.id` 或等价字段建立关联
- `getLinkedRiskActions(projectId)` 是 Web 侧读取入口

iOS 本轮允许的策略：

- **直接用 Supabase 读取同一批底层表/关联关系，构建 read-only foundation**

禁止：

- 伪造写入按钮
- 伪造“转为风险动作”已支持
- 为了偷懒改成静态 mock 数据

如果 Web 的 linked action 关系链无法被当前 iOS 侧安全、稳定读取，必须如实记录并终止该方向，改为最小修复 prompt，而不是硬写伪实现。

### 3.B UI scope

本轮只允许在 `ProjectDetailView` 内新增一个 foundation 区块，例如：

- `Linked Risk Actions`

允许展示的最小信息，以 Web source-of-truth 为准，预计包括：

- action title
- status
- owner / assignee（若可安全读到）
- priority / due date（若 Web 确有展示且当前关系链稳定）
- 最多前 3 条 + more 提示（若符合现有 detail 信息密度）

不允许扩成完整 risk actions module。

### 3.C 状态机

至少要有以下状态：

1. `idle`
2. `loading`
3. `noRiskAnalysisSource`
   - 当前项目还没有对应 risk analysis / 没有 `summaryId` / 没有可关联源
4. `empty`
   - 风险分析存在，但没有 linked actions
5. `success`
6. `failure`
7. `denied`

必须显式建模，不允许把多个语义混成一个 `nil`。

### 3.D 错误隔离

linked actions 必须有自己的独立状态，至少不要污染：

- `errorMessage`
- `summaryErrorMessage`
- `riskAnalysisErrorMessage`
- `deleteErrorMessage`
- `enrichmentErrors`

失败时如果已经拿到旧快照，优先保留旧快照。

### 3.E 视觉与交互

- 复用现有 Projects detail card 视觉语言
- 不新增花哨动效
- 不引入 modal
- 不改 detail 主布局骨架
- 只在 detail 内做增量延展

### 3.F 诚实文案

必须明确说明：

- linked risk actions 当前为只读展示
- “转为风险动作”写入动作仍在 Web 或后续轮次

不能让用户误以为 iOS 已支持同步生成/写入。

### 3.G `.projects` 状态

本轮完成后：

- **`.projects` 仍保持 `.partial`**

因为仍至少缺：

- resolution feedback
- generate-risk-from-iOS
- summary LLM narrative parity
- 其他更深层项目 parity 缺口
