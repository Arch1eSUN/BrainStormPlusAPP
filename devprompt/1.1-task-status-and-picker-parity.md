## 3. 目标 A — Task status enum parity

当前已知差异：

- Web: `todo`, `in_progress`, `review`, `done`
- iOS: `todo`, `in_progress`, `in_review`, `done`, `canceled`

本轮要求：

1. 明确核对 Web source of truth 的实际允许值。
2. 判断 iOS 是否必须改 enum rawValue，还是通过 adapter 映射层兼容。
3. 不能只改显示文案，必须对齐真实数据库读写语义。
4. 如果 `canceled` 在 Web 端并无稳定支持，不得继续把它当作完全对齐状态默许通过。
5. 优先选择**最小、可回滚、对现有 create/update 影响最小**的方案。

验收要求：

- 说明最终采用的对齐策略
- 扫描/源码可证实 status 写入不会再制造 Web/iOS 语义漂移

## 4. 目标 B — assignee / project picker parity assessment + implementation

当前真实状态：

- iOS create flow 没有 assignee/project picker UI
- create fallback 使用 authenticated user 作为 owner/reporter/creator/assignee

本轮要求：

1. 先核 Web create flow 是否真的要求 project/owner 选择器。
2. 如果数据源与交互依赖已具备，则补齐最小可用 picker。
3. 如果依赖明显缺失、会把本轮扩散成更大 scope，则必须：
   - 明确记录为什么本轮不能完整做
   - 给出 source-of-truth 证据
   - 落成不阻塞的最小兼容实现或明确 deferred debt
4. 不允许凭猜测硬塞假 picker。
