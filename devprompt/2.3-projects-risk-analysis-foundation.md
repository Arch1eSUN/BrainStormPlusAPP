## 3. 目标

### 目标 A — 先核清 Web risk analysis 的真实 source of truth

必须先独立确认 Web 上 Projects risk analysis 的真实入口与真实数据路径。

至少回答清楚：

1. Web 的 risk analysis 是否存在真实按钮 / 面板 / section？
2. 它走 server action、API route，还是直接 client fetch？
3. 输入是什么：project row、tasks、daily logs、cached risk summary、risk summary id，还是别的？
4. 输出 shape 是什么：summary string、riskLevel、summaryId、meta，还是别的？
5. 失败时 Web 如何表现？
6. 缓存 / persistence 是否存在？如果存在，落在哪张表？

如果 Web risk analysis 只是 placeholder / dead code / vestigial type：
- 必须如实记录 source-of-truth discrepancy
- 不得假装 parity 已存在
- 可以做最小 foundation，但必须在 ledger 里明确写清“Web 当前真实状态”

### 目标 B — iOS Detail Risk Analysis Foundation

仅在 `ProjectDetailView` 路径上补一个**最小可审计**的 risk analysis foundation。

最低要求：

- 有明确入口（按钮或 section 内触发）
- 有 loading state
- 有 success state
- 有 isolated failure state
- 成功后用户能在 detail 页面看到 risk summary 与 risk level
- 不阻断原有 owner / tasks / daily / weekly / summary 内容
- 不做大 UI 改版

foundation 允许的最小形态示例：

- `Analyze risk` / `Refresh risk analysis` button
- 点击后请求 Web 对应 risk-analysis action / endpoint；若 iOS 无法安全直连，则必须记录 discrepancy 并交付最小可审计替代 foundation
- 在 detail 页 read-only 展示一段 risk summary 文本与 risk level badge
- 如 Web 存在缓存元数据（generatedAt / model / scenario / isCached），则优先对齐最小必要元数据展示

### 目标 C — State truth / failure isolation

新增 risk analysis state 必须与以下错误面隔离：

- `errorMessage`（detail first-load / gate / project fetch）
- `enrichmentErrors[.tasks|.dailyLogs|.weeklySummaries]`
- `deleteErrorMessage`
- `summaryErrorMessage`
- list fetch / owner hydrate / task-count surfaces
- edit/member surfaces

最低要求：

- risk analysis 请求失败不得让 detail 页整体报错
- risk analysis 失败不得抹掉已加载的 project / tasks / daily / weekly / summary 内容
- risk analysis loading 不得阻断 delete / edit / navigation 主路径
- risk analysis state 不得借用 2.2 的 summary state 伪装实现

### 目标 D — Scope control

本轮可以做：

- Web risk analysis source re-read
- iOS detail risk-analysis request state
- read-only risk summary UI
- risk level badge
- 最小 DTO / response decode
- 如 Web 存在 cached meta，可做最小 generated-at / cached badge 对齐
- ledger update

本轮不要做：

- linked risk actions
- resolution feedback
- 将 risk item 同步成 risk action
- schema changes
- long-term persistence / offline cache
- create/edit/delete/member redesign
- task-count redesign
- task CRUD redesign
- AI multi-step workflow orchestration
- streaming chat UI
- prompt engineering backend rewrite
- analytics / telemetry system
- Web-side route / Edge Function 新增
