# 1.6 — Projects Detail Membership Gate + Ordering Alignment Foundation

**执行模型：Claude Opus 4.6**

## 0. 轮次定位

`1.5 Projects Server-Side Filter + Membership Scope Foundation` 已通过 Winston 正式审计：

- `docs/parity/26-winston-audit-1.5.md`

本轮进入 `1.6`，继续收口 **Projects**，但仍然只做 **foundation 级安全/语义补强**，不扩散到 full detail parity / CRUD / members UI / AI parity。

`1.5` 已交付：

- Projects list 的 server-side `search + status` filter foundation
- Projects list 的 membership scope foundation
- admin / non-admin / no-membership 三种 list 语义收口

但 `1.5` 审计后，Projects 还剩一个更高优先级的真实缺口：

1. **iOS Project detail 仍没有 membership gate**
2. **Web `fetchProjectDetail()` 对 non-admin non-member 会拒绝访问，但 iOS detail 仍只是按 `project.id` 直取**
3. **Projects list 的排序仍与 Web 不一致：iOS `updated_at DESC`，Web `created_at DESC`**

所以 `1.6` 的目标不是做更多 detail 内容，而是优先把 **detail access semantics** 与 **list ordering semantics** 再向 Web source of truth 收口。

不要扩展到：

- Projects create / update / delete
- member picker / member management UI
- owner profile full rendering UI
- tasks / daily logs / weekly summaries detail enrichment
- AI summary / risk analysis / linked actions / resolution feedback
- OKR / Deliverables / Approvals / Admin 等跨模块工作

## 3. 目标

### 目标 A — Project detail membership gate foundation

把 `ProjectDetailViewModel.fetchDetail()` 从“只按 `project.id` 拉单行”推进到更接近 Web source of truth 的 **detail membership gate foundation**。

目标对齐 Web `fetchProjectDetail(projectId)`：

- Admin+：可读取 detail
- 非 Admin 且是 member：可读取 detail
- 非 Admin 且不是 member：不可读取 detail

实现要求：

1. 必须先核对 Web `fetchProjectDetail()` 如何做 membership check：
   - `project_members`
   - `project_id`
   - `user_id`
   - `maybeSingle()` / no membership -> denied
2. iOS 侧可以不复刻 Web server action 的全部结构，但必须在真实查询语义上实现**同等 access gate**。
3. 不要把 detail gate 做成假的 UI gate（例如仅依赖前一页传参而完全不查 membership）。
4. 如果 detail gate 需要传入 `role` / `userId` / `currentProfile`，必须选一个最小、真实、可复用的接入点。
5. 如果当前 detail 打开时尚未拿到 profile/userId，必须采用**更保守**的处理，而不是放行读取。
6. 禁止因为赶进度而直接说“依赖 RLS 就够了”并跳过 app-side foundation 实现。

可以接受：

- detail gate 失败时显示明确 error / denied state
- 继续保留 1.4 的 seeded row rendering，在 fetch denied 后转为受控错误态

但不要接受：

- non-member 仍能在 app-side 直接读取 detail row
- 仅在 list 做 gate，detail 完全不做 gate

### 目标 B — Project list ordering alignment foundation

把 `ProjectListViewModel` 的 Projects 排序从当前的：

- `updated_at DESC`

对齐到 Web 当前 source of truth：

- `created_at DESC`

实现要求：

1. 必须先核对 Web `fetchProjects()` 当前排序字段。
2. 改动必须同时适用于：admin path / membership-scoped path。
3. 若因数据空值或 SDK 限制无法完全对齐，必须记录真实差异到 ledger，不能假装一致。

### 目标 C — ViewModel / state truth

围绕 1.6 的 detail access semantics 调整 `ProjectDetailViewModel` 与必要的 `ProjectDetailView` / `ProjectListView` 接线，最小要求：

- `isLoading` / `errorMessage` 保持可用
- denied state 与普通网络错误尽量不要语义混淆
- 仍然保留 1.4/1.5 已有可工作的 detail foundation
- 不要把 list semantics 搞坏
- 不要把 `.projects` 路由打回 backlog

可以接受：

- 在 detail view 中新增受控 denied banner / denied empty state
- 在必要时把 identity 注入从 list 传到 detail view model 或 detail fetch helper

但不要接受：

- 为了实现 gate 重写整套 Projects navigation
- 为了对齐 Web detail gate 顺手扩展 tasks / daily / weekly / AI sections

### 目标 D — Ledger truth

更新：

- `findings.md`
- `progress.md`
- `task_plan.md`
- 如有必要新增 `docs/parity/27-winston-ready-1.6-notes.md`

必须明确写：

- detail gate 如何落地
- admin / member / non-member detail 三种语义如何处理
- list ordering 是否已从 `updated_at` 对齐到 `created_at`
- 仍缺哪些 Projects parity：detail enrichment、owner profile join、CRUD、members UI、AI、risk 等
- `.projects` 是否仍应保持 `.partial`

默认预期：**`.projects` 仍保持 `.partial`**，除非你能基于真实代码和 Web 对照证明已经完成到可升格的程度。大概率本轮不应升格。
