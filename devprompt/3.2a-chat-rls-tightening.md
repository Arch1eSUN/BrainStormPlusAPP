# 3.2a — Chat RLS Tightening (Cross-Stack Security Sprint)

**执行模型：Claude Opus 4.7（主路径）+ Sonnet 4.6 subagent（机械迁移 task）**

## 0. 轮次定位

3.1 Team Chat Foundation 通过 Winston 独立审计 PASS（有条件），遗留 **1 条系统级安全债**：`chat_channels` / `chat_messages` 的 SELECT RLS 仍是 `USING (true)`，意味着任何拿到有效 JWT 的客户端——包括未来被越狱的 iOS 设备、浏览器 DevTools 直连 Supabase 的任意终端——都能读取**全部频道元信息和全部消息正文**，不管他是否属于该频道。

3.1 应对方案是**客户端复刻** `getAccessibleChannelMap` / `ensureChannelAccess`，所以**渲染层不会暴露**越界数据。但这只是遮羞布：只要攻击者跳过我们的客户端（用 `supabase-js` 裸调、用 curl + JWT、写自己的 app 壳），数据库会**忠实返回全量**。这是 iOS 新增的攻击面——Web 因为所有 chat action 走 `createAdminClient()` 服务端执行，客户端根本拿不到数据库句柄，所以 `USING (true)` 在 Web-only 时代没问题；3.1 开始 iOS 直连 Supabase 后，安全边界被破。

**本轮唯一目标**：把 SELECT 策略从 `USING (true)` 收紧到成员可见域，让 RLS 自己变成授权源，iOS 客户端访问控制从"安全担当"降级成"渲染优化"。

**明确不做**：
- 新功能（附件、回复、撤回、表情、mentions、搜索、新建会话）全部推给 3.2 及之后的 feature sprint。
- iOS UX 改动——collapse 客户端三段联合是可选优化（见 Task D），不是必须。

### 1.4 必须确认的事实

- **Supabase auth.uid() 在 RLS policy 内对 service-role 连接返回 NULL**——`createAdminClient()` 绕过 RLS 的机制不变，所以 Web server action 不受影响的前提是：**所有读路径**都走 admin client。必须用 grep 验证不存在"用户 client 读 chat 表"的调用点。
- **Realtime RLS 语义**：Supabase Realtime 会在每条 change event 上以**当前连接的 user 身份**重放 SELECT policy，不满足策略的 event 会被 drop。这意味着收紧 SELECT 后，iOS / Web 用户只会收到自己有权见的频道的消息——这**是我们想要的**，但也意味着如果 policy 写错，**Realtime 会静默丢消息**，极难 debug。必须在验收阶段手动覆盖 Realtime 场景。
- **`chat_channel_members.user_id = auth.uid()` 是收紧 SELECT 的关键 predicate**——但 announcement 频道需要额外分支（全员可见），否则 announcement 会对非成员不可见。

## 2. 地面真相：当前策略 vs 目标策略

### 2.1 当前（3.1 完成时）

```sql
-- schema.sql:375
CREATE POLICY "chat_channels_select" ON chat_channels
  FOR SELECT USING (true);

-- schema.sql:347-348
CREATE POLICY "chat_messages_select" ON chat_messages
  FOR SELECT USING (true);
```

**实际效果**：任何带有效 JWT 的连接读到全表。Web 靠 server action 前置 gate；iOS 靠客户端 `getAccessibleChannelMap` / `ensureChannelAccess` 复刻。Winston 审计认定：系统级安全债。

### 2.2 目标（3.2a 完成后）

```sql
-- chat_channels_select: announcement ∪ owned ∪ membership
CREATE POLICY "chat_channels_select" ON chat_channels
  FOR SELECT USING (
    type = 'announcement'
    OR created_by = auth.uid()
    OR EXISTS (
      SELECT 1 FROM chat_channel_members m
      WHERE m.channel_id = chat_channels.id
        AND m.user_id = auth.uid()
    )
  );

-- chat_messages_select: messages 的可见性 = 所在频道的可见性
CREATE POLICY "chat_messages_select" ON chat_messages
  FOR SELECT USING (
    EXISTS (
      SELECT 1 FROM chat_channels c
      WHERE c.id = chat_messages.channel_id
        AND (
          c.type = 'announcement'
          OR c.created_by = auth.uid()
          OR EXISTS (
            SELECT 1 FROM chat_channel_members m
            WHERE m.channel_id = c.id
              AND m.user_id = auth.uid()
          )
        )
    )
  );
```

**等价性保证**：
- `chat_channels` 新策略的三段联合**与 Web `getAccessibleChannelMap`（chat.ts:252-282）逻辑完全对齐**——这是刻意设计，让 RLS 成为 Web 已验证逻辑的 DB 镜像。
- `chat_messages` 新策略 = "频道可见" + "消息 ∈ 频道"，保证不会出现"频道看得到但消息读不了"或反过来的撕裂。

**不动的策略**：INSERT（`auth.uid() = sender_id`）、UPDATE（023 / 026）、chat_channel_members（028）全部保持不变。本轮只动 SELECT。

### 2.3 Web server action 的影响面

**理论上为零**——因为所有读路径走 `createAdminClient()`，而 service-role 连接 bypass RLS。但必须**实测验证**（Task B），不能只靠推理：

- 任意历史 commit 里有没有遗留的 `createSupabaseServerClient()` / `createClient()` 直读 chat 表？
- `use-realtime.ts` 在浏览器端走的是 user-role JWT，收紧 SELECT 会改变**它**的可见消息集。这是**期望行为**（Web 浏览器端本来就不该看到非成员频道的新消息），但要确认 `dashboard/chat/page.tsx` 的 UI 在策略收紧后仍正确显示——它本来就有 server-side `fetchChannels` 过滤，但要验证 Realtime 收不到的那部分消息不会让 UI 卡住。

## 3. 任务清单

### Task A — 写迁移

**文件**：`BrainStorm+-Web/supabase/migrations/029_chat_rls_tightening.sql`（新建，版本号紧接 028）

**内容**：
```sql
-- Sprint 3.2a: Tighten chat SELECT RLS from USING (true) to membership-scoped.
-- Motivation: 3.1 introduced iOS direct-to-Supabase reads; the blanket
-- USING (true) SELECT on chat_channels and chat_messages means any holder
-- of a valid user JWT can read the full tables. Collapse the visibility
-- predicate to the same membership union Web's getAccessibleChannelMap
-- (src/lib/actions/chat.ts:252-282) has been enforcing at the application
-- layer. Web server actions using createAdminClient() are unaffected
-- because service-role connections bypass RLS; but every user-role
-- connection (browser Realtime, iOS direct reads) is now DB-authoritative.

BEGIN;

-- Replace chat_channels SELECT
DROP POLICY IF EXISTS "chat_channels_select" ON chat_channels;
CREATE POLICY "chat_channels_select" ON chat_channels
  FOR SELECT
  USING (
    type = 'announcement'
    OR created_by = auth.uid()
    OR EXISTS (
      SELECT 1 FROM chat_channel_members m
      WHERE m.channel_id = chat_channels.id
        AND m.user_id = auth.uid()
    )
  );

-- Replace chat_messages SELECT
DROP POLICY IF EXISTS "chat_messages_select" ON chat_messages;
CREATE POLICY "chat_messages_select" ON chat_messages
  FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM chat_channels c
      WHERE c.id = chat_messages.channel_id
        AND (
          c.type = 'announcement'
          OR c.created_by = auth.uid()
          OR EXISTS (
            SELECT 1 FROM chat_channel_members m2
            WHERE m2.channel_id = c.id
              AND m2.user_id = auth.uid()
          )
        )
    )
  );

COMMIT;
```

**索引审查**：`chat_channel_members(user_id, channel_id)` 必须已存在（028 migration 已建索引 `idx_chat_channel_members_user_id`，覆盖 policy 内的 `m.user_id = auth.uid()` 查询——验证 migration 链确认）。

**验证**：
- 本地 `supabase db reset` 成功重放迁移链。
- Policy SQL 在 Supabase Studio 可见且启用。
- 没有任何 INSERT / UPDATE / DELETE 策略变化（grep schema dump 对比）。

**子任务 B2（subagent 2）**：Realtime 订阅路径核查。
- `src/hooks/use-realtime.ts` + `src/app/dashboard/chat/page.tsx` 的完整消费流程

**汇总交付**：一个 checklist markdown，每个读点打 ✅ 不受影响 / ⚠️ 受影响但安全 / ❌ 破坏。

### Task C — Shadow 验证（dev + staging）

**前提**：B1/B2/B3 全绿。

**C1 — Dev 本地验证**（`supabase start` + 本地数据库）：
- seed 三个 profile（userA、userB、userC），两个 channel：
  - `dm-AB`（group/direct，members: A、B）
  - `team-announce`（announcement，无显式 member）
- 以 A 身份带 JWT 走 `supabase-js` 直连：
  - `SELECT * FROM chat_channels` → 应返回 `dm-AB` + `team-announce`，**不含 B 和 C 的其他私聊**
  - `SELECT * FROM chat_messages WHERE channel_id = <C 的某私聊>` → 返回空
  - `SELECT * FROM chat_messages WHERE channel_id = dm-AB` → 返回全部
- 以 C 身份同样三条：第一条应仅 `team-announce`，第二/三条均应**返回 dm-AB 的消息为空**。
- INSERT / UPDATE 路径覆盖：A 向 `dm-AB` 插入消息成功；C 向 `dm-AB` 插入消息被 RLS 拒（INSERT 策略不变，依然走 `auth.uid() = sender_id` 通过，但 **Realtime broadcast 对 C 不可见**——这是期望行为，注记）。

**C2 — iOS 端到端 smoke**（3.1 chat views 直接对接新策略）：
- 用现有 3.1 iOS app 对准 dev DB：登录 A，看到两个频道；登录 C，只看到 announcement；验证 A 发消息 B 收到 realtime push；验证 C 发不进 dm-AB。
- **关键**：3.1 客户端仍保留 `getAccessibleChannelMap` 客户端复刻——**不要在 C1/C2 阶段删除**，要确认"RLS 独立授权"和"客户端复刻"两层都工作。Task D 才处理客户端简化。

**C3 — Staging shadow 观察期（最小 24h）**：

- 前端 sentry 观察是否有 `fetchChannels` / `fetchMessages` 返回空白但 UI 卡 loading 的 case。

**验证硬门**：

- 24h shadow 无新增 RLS-related error。

### Task D — iOS 客户端访问控制简化（**可选，建议推**）

**前提**：C 全绿 + 产线上线后 ≥ 48h 无 regression。

**动机**：RLS 收紧后，iOS 客户端的 `getAccessibleChannelMap` 三段联合和 `ensureChannelAccess` 已经**退化成冗余渲染优化**——DB 本身只返回可见数据。保留客户端逻辑会增加维护成本（未来新增访问规则要改两遍），但在"RLS 是唯一真相"还没被独立审计 green-lit 前，双层防御**不是坏事**。

**两个选项**：

- **选项 D1（保守，推荐）**：保留客户端三段联合，**加一条注释**说明现在它是渲染层优化，不是安全担当；所有访问控制验证在 DB 层。

**本轮建议执行 D1**——D2 需要 Winston 再审一轮 + 一个独立 sprint 决议，不在本轮 scope。
