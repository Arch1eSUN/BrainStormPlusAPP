# 2.5 — Projects Resolution Feedback Foundation

**执行模型：Claude Opus 4.6**

## 0. 轮次定位

`2.4 Projects Linked Risk Actions Foundation` 已通过 Winston 独立审计：

- `docs/parity/44-winston-audit-2.4.md`

当前 Projects 模块仍保持 `.partial`。在 2.4 已完成 linked risk actions read-only foundation 后，下一项最自然、最窄、最可审计的 detail-surface gap 是：

- **Projects Resolution Feedback Foundation**

本轮只做：

- 读取并展示与项目风险分析关联的 resolution feedback foundation
- 仅限 detail 页面 risk analysis / linked actions 区块向下延展的 read-only foundation
- 不做 risk action 创建/编辑/关闭
- 不做 “转为风险动作” 写路径
- 不做 iOS 侧 generate-risk
- 不做 Web-side `/api/ai/project-risk` HTTP endpoint
- 不做任务、成员、删除、AI summary、task-count redesign

## 2. 目标

在 iOS `ProjectDetailView` 中，为 2.4 的 linked actions 区块继续增加一个 **Resolution Feedback** foundation 区块，实现以下能力：

1. **读取与当前项目风险分析相关的 resolution feedback 聚合信息**。
2. 只做 **read-only 展示**。
3. 明确区分：
   - 还没检查
   - 没有风险分析 source
   - 有 source 但没有 resolution feedback 数据
   - 读取失败
   - 读取成功
4. 保持诚实文案：
   - iOS 当前只读，不支持写 resolution / governance actions。
5. 严格隔离错误面：
   - resolution feedback 失败不能污染 detail 主态 / risk analysis / linked actions / AI summary / enrichment / delete。

---

## 3. 实现约束

### 3.A 数据来源与 source of truth

必须先基于 Web 重新确认真实链路。

预期真实情况（必须重新验证，不可照抄本段当事实）：

- `getProjectRiskResolutionSummary(projectId)` 基于 `project_risk_summaries.id` 与 `risk_actions.ai_source_id` 关系聚合
- 返回类似：
  - total
  - resolved
  - dismissed
  - active
  - followUpRequired
  - dominantCategory
  - recentResolutions
  - reopenedCount

本轮允许策略：

- **直接用 Supabase 读取同一批底层表/关联关系，构建 read-only foundation**

禁止：

- 伪造聚合数字
- 伪造 governance intervention 已支持
- 用静态 mock 顶替真实链路

如果 Web 的 resolution feedback 真实链路当前无法被 iOS 安全、稳定读取，必须如实记录并终止该方向，改为最小修复 prompt，而不是硬写伪实现。

### 3.B UI scope

本轮只允许在 `ProjectDetailView` 内新增一个 foundation 区块，例如：

- `Resolution Feedback`

允许展示的最小信息，以 Web source-of-truth 为准，预计包括：

- total / resolved / dismissed / active 计数
- follow-up required
- dominant category
- recent resolutions（若 Web 有且当前关系链稳定）
- reopened count（若 Web 有且当前关系链稳定）

不允许扩成完整 risk action management module。

### 3.C 状态机

至少要有以下状态：

1. `idle`
2. `loading`
3. `noRiskAnalysisSource`
4. `empty`
5. `loaded`
6. `failure`
7. `denied`

必须显式建模，不允许把多个语义混成一个 `nil`。

### 3.D 错误隔离

resolution feedback 必须有自己的独立状态，至少不要污染：

- `errorMessage`
- `summaryErrorMessage`
- `riskAnalysisErrorMessage`
- `linkedRiskActionsErrorMessage`
- `deleteErrorMessage`
- `enrichmentErrors`

失败时如果已经拿到旧快照，优先保留旧快照。

### 3.E 视觉与交互

- 复用现有 Projects detail card 视觉语言
- 不新增花哨动效
- 不引入 modal
- 不改 detail 主布局骨架
- 只在 detail 内做增量延展

### 3.F 诚实文案

必须明确说明：

- resolution feedback 当前为只读展示
- governance intervention / resolution write path 仍在 Web 或后续轮次

不能让用户误以为 iOS 已支持处理风险动作生命周期。

### 3.G `.projects` 状态

本轮完成后：

- **`.projects` 仍保持 `.partial`**

因为仍至少缺：

- “转为风险动作” sync 写路径
- generate-risk-from-iOS
- summary LLM narrative parity
- 其他更深层项目 parity 缺口
